# Story: Internationalisation (i18n)

## User Story

> As a **German-speaking player**, I want **the app's interface to appear in German**
> so that **I can play comfortably in my native language without having to parse English UI.**
>
> As an **Android user**, I want **to set the app's language independently from my device language**
> so that **I can keep my phone in English but play the game in German, or vice versa.**
>
> As an **iOS user**, I want **to set the app's language from the iOS Settings page for the app**
> so that **I can choose a different language for this game without changing my entire device locale.**

---

## Background

Hulls & Hellfire currently ships all copy in English. German is the first additional locale — both because it is a high-value mobile market and because English/German is an instructive first pair: similar UI density, Latin script, no right-to-left complexity, but meaningful divergence in plural forms, word length, and register.

The feature has two orthogonal concerns:

1. **Translation layer** — loading and rendering locale-specific strings throughout the app.
2. **Locale detection** — knowing which language to display, respecting the device locale, the iOS per-app language (iOS 16+), the Android per-app language (Android 13+), and the user's explicit in-app selection. English is the fallback whenever the resolved locale is not `en` or `de`.

---

## Scope

| Area | Included |
|---|---|
| All visible UI strings (buttons, labels, verdicts, tutorial copy, stats) | Yes |
| English (`en`) locale | Yes (current strings become the baseline) |
| German (`de`) locale | Yes |
| Android Per-app Language Preferences (Android 13+) | Yes |
| In-app language picker (Settings screen) | Yes |
| iOS per-app language (Settings → [App] → Language) | Yes, via `expo-localization` |
| Text images containing localizable copy (`commence_firing.png`) | Yes |
| Locale-aware number/date formatting | Out of scope (no dates; ship counts are numerals) |
| Right-to-left layout | Out of scope |
| Additional locales beyond `en` / `de` | Out of scope |

---

## Locale Detection

Locale resolution follows this priority chain (highest wins):

1. **User's in-app selection** — stored in persistent storage, overrides everything.
2. **iOS per-app language** (iOS 16+) — set in **Settings → [App] → Language**, surfaced by `expo-localization`'s `getLocales()`.
3. **Android per-app language** (Android 13+) — set in **Settings → System → Language → App Language**, also surfaced by `getLocales()`.
4. **Device locale** — the system's active language, also from `getLocales()`.
5. **Fallback** — `en`. Any device locale that is neither `en` nor `de` resolves to English.

`expo-localization` is the only package needed for detection. Its `getLocales()` call returns per-app locale overrides on both platforms before falling back to the device locale — no additional native module is required.

## Android Per-app Language Preferences

Android 13 (API 33) introduced `LocaleManager`, which lets users set a per-app language in **Settings → System → Language → App Language**. For this to appear in the system UI, the app must declare its supported locales in a locale config resource.

### Required additions

**`app.json` / Expo config plugin:**

```json
{
  "android": {
    "localeConfig": {
      "path": "./android/app/src/main/res/xml/locales_config.xml"
    }
  }
}
```

**`res/xml/locales_config.xml`** (generated or committed):

```xml
<?xml version="1.0" encoding="utf-8"?>
<locale-config xmlns:android="http://schemas.android.com/apk/res/android">
  <locale android:name="en"/>
  <locale android:name="de"/>
</locale-config>
```

**`AndroidManifest.xml`** (via config plugin or direct edit in bare workflow):

```xml
<application
  android:localeConfig="@xml/locales_config"
  ...>
```

Once declared, `expo-localization`'s `getLocales()[0].languageTag` will return the per-app locale the user selected in system settings — no extra API calls needed.

## iOS Per-app Language Preferences

iOS 16 introduced per-app language selection in **Settings → [App] → Language**. iOS automatically surfaces this UI for any app that declares its supported locales in `Info.plist` via the `CFBundleLocalizations` key.

### Required additions

**`app.json`:**

```json
{
  "ios": {
    "infoPlist": {
      "CFBundleLocalizations": ["en", "de"],
      "CFBundleDevelopmentRegion": "en"
    }
  }
}
```

`CFBundleDevelopmentRegion` marks English as the development language, which iOS uses as the fallback when no translation is found for the active locale. No additional config plugin or native change is needed — Expo writes these values into `Info.plist` at build time.

Once declared, `expo-localization`'s `getLocales()[0].languageTag` returns the per-app locale chosen in iOS Settings, indistinguishable from a device-locale change — the resolution chain handles both uniformly.

## Translation Layer

### Library: `i18next` + `react-i18next`

**Why i18next:**
- TypeScript-first with full type inference on translation keys (zero `any` needed).
- Robust plural handling — German has irregular plural forms; i18next's `count`-based interpolation handles them correctly out of the box.
- Namespace support for splitting large translation files (useful when tutorial copy grows).
- No native dependencies — works cleanly in Expo managed workflow.
- Largest ecosystem of any JS i18n library; well-maintained in 2025.

**Packages:**

```
i18next
react-i18next
expo-localization   ← already used for locale detection above
```

### Translation file structure

```
locales/
  en/
    common.json       ← shared labels, buttons, nav
    battle.json       ← in-battle verdicts, turn indicators
    tutorial.json     ← all tutorial overlay copy
    stats.json        ← stats screen labels
  de/
    common.json
    battle.json
    tutorial.json
    stats.json
```

### Typed translation keys

Generate a `resources` type from the `en` locale so TypeScript catches missing or misspelled keys at compile time:

```ts
// i18n/index.ts
import en from '@/locales/en/common.json';

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'common';
    resources: { common: typeof en };
  }
}
```

Extend with `battle`, `tutorial`, and `stats` namespaces as they are wired up.

### Initialisation

```ts
// i18n/index.ts
import i18next from 'i18next';
import { initReactI18next } from 'react-i18next';
import { getLocales } from 'expo-localization';

import enCommon from '@/locales/en/common.json';
import deCommon from '@/locales/de/common.json';
// ... other namespaces

const deviceLocale = getLocales()[0]?.languageCode ?? 'en';
const savedLocale = /* read from AsyncStorage */ null;
const resolvedLocale = savedLocale ?? (deviceLocale === 'de' ? 'de' : 'en');

i18next
  .use(initReactI18next)
  .init({
    lng: resolvedLocale,
    fallbackLng: 'en',
    resources: {
      en: { common: enCommon },
      de: { common: deCommon },
    },
    interpolation: { escapeValue: false },
  });

export default i18next;
```

### Usage in components

```tsx
import { useTranslation } from 'react-i18next';

function BattleVerdict({ result }: { result: 'hit' | 'miss' | 'sunk' }) {
  const { t } = useTranslation('battle');
  return <Text style={styles.verdict}>{t(`verdict.${result}`)}</Text>;
}
```

## In-app Language Picker

A language picker lives on the Settings screen (when built) and eventually also in an early onboarding step. It presents exactly two options: **English** and **Deutsch**.

- On selection, the new locale is written to persistent storage (`language_preference: 'en' | 'de'`) and `i18next.changeLanguage(locale)` is called.
- The UI re-renders immediately — no restart required.
- On Android 13+, the in-app picker and the system Settings entry are independent. Either can change the active locale; the in-app selection takes priority per the resolution chain above.

## In-game Text Inventory

All visible strings must be translated. Key areas:

| Namespace | Examples |
|---|---|
| `common` | button labels (DEPLOY FLEET, RETREAT, PLAY AGAIN), nav items, Settings labels |
| `battle` | HIT, MISS, SUNK, VICTORY, DEFEAT, turn indicator text, retreat confirmation |
| `tutorial` | All callout titles and body copy from both tutorial sequences |
| `stats` | Column headers, stat labels, empty-state message |

Ship names (`Carrier`, `Battleship`, etc.) are treated as proper nouns and left untranslated — they appear in the German build in English as naval jargon.

---

## Visual Design Considerations

German strings are consistently longer than English equivalents (typically 20–40% more characters). All UI components must accommodate expansion without clipping or overflow:

- **Button labels:** Minimum touch target must remain ≥ 44px tall; width grows with text — test DEPLOY FLEET → FLOTTE EINSETZEN.
- **Verdicts (HIT / MISS / SUNK):** These already use large BlackOpsOne; German equivalents (TREFFER / VERFEHLT / VERSENKT) are longer — ensure the verdict banner does not clip on smaller screens.
- **Tutorial callouts:** Body copy wraps naturally; verify callout card height doesn't push below the screen edge on short devices.
- **Stats labels:** Monospace readout column may need a slightly wider label column in German.

No layout changes needed for RTL — neither locale requires it.

## Text Images

Some assets are rasterised images of text rather than rendered strings, and must be swapped out per locale. The game title (`IMAGES.title`) is a brand wordmark and stays in English for all locales. The following images contain localizable copy:

| Key | Current text | German replacement |
|---|---|---|
| `IMAGES.commenceFiring` | COMMENCE FIRING | FEUER FREI |

### File structure

Locale-specific variants live under `assets/images/locales/`:

```
assets/images/locales/
  en/
    commence_firing.png
  de/
    commence_firing.png
```

The existing `assets/images/commence_firing.png` becomes the `en` variant (moved, not deleted).

### Locale-aware image map

Add a `LOCALE_IMAGES` export to [constants/assets.ts](constants/assets.ts) alongside the existing `IMAGES` map:

```ts
export const LOCALE_IMAGES = {
  en: {
    commenceFiring: require('@/assets/images/locales/en/commence_firing.png'),
  },
  de: {
    commenceFiring: require('@/assets/images/locales/de/commence_firing.png'),
  },
} satisfies Record<'en' | 'de', { commenceFiring: unknown }>;
```

### Usage in components

```tsx
import { useTranslation } from 'react-i18next';
import { LOCALE_IMAGES } from '@/constants/assets';

function CommenceFiringFlash() {
  const { i18n } = useTranslation();
  const locale = (i18n.language === 'de' ? 'de' : 'en') as keyof typeof LOCALE_IMAGES;
  return (
    <Image source={LOCALE_IMAGES[locale].commenceFiring} style={styles.flashImage} resizeMode="contain" />
  );
}
```

Adding a new locale-specific image in the future follows the same pattern: add the file under `assets/images/locales/<locale>/`, add the `require` to `LOCALE_IMAGES`, and TypeScript's `satisfies` constraint will enforce that every locale covers every key.

## Persistence

```
language_preference: 'en' | 'de' | null   ← null means "follow device"
```

Stored in `AsyncStorage`. Read once at app start during i18n initialisation. Updated immediately on in-app language change.

## Acceptance Criteria

- [ ] All visible UI strings are rendered via `t()` — no hard-coded English strings remain in components.
- [ ] The app launches in German when the device locale is `de` and no in-app preference has been saved.
- [ ] The app launches in English for any non-German device locale (fallback).
- [ ] The in-app language picker changes the active locale immediately without a restart.
- [ ] The user's in-app language selection persists across app restarts.
- [ ] On iOS 16+, the app appears in **Settings → [App] → Language** with English and German as options.
- [ ] Selecting a language via iOS Settings is respected on next app launch.
- [ ] On Android 13+, the app appears in the system **App Language** settings entry with English and German as options.
- [ ] Selecting a language via the Android system Settings entry is respected on next app launch (or immediately if the app is still running, via `expo-localization`'s live locale updates).
- [ ] The in-app selection takes priority over both the iOS and Android system selections.
- [ ] Any device locale other than `de` resolves to English — no unsupported locale is ever displayed.
- [ ] `IMAGES.commenceFiring` is replaced by `LOCALE_IMAGES[locale].commenceFiring` at its call site — the old key is removed from `IMAGES`.
- [ ] The German `commence_firing.png` variant ("FEUER FREI") renders at the same dimensions as the English original.
- [ ] Adding a new locale-specific image to `LOCALE_IMAGES` without providing it for every locale causes a TypeScript error.
- [ ] German battle verdicts (TREFFER / VERFEHLT / VERSENKT) are not clipped on a 375px-wide screen.
- [ ] German button labels (FLOTTE EINSETZEN, ZURÜCKZIEHEN) are not clipped at default font sizes.
- [ ] TypeScript compilation fails if a translation key referenced in code does not exist in the `en` locale files.
- [ ] No `any` casts are introduced to satisfy the i18next type definitions.
- [ ] `expo-localization` is the sole package responsible for reading the device/per-app locale — no raw `NativeModules` calls.

---

## Out of Scope

- Third locale (French, Spanish, etc.)
- Locale-specific date or number formatting
- Right-to-left layout support
- Dynamic loading of translation files from a remote CMS
- Translator tooling / translation memory (Phrase, Lokalise, etc.)
- Automated screenshot testing per locale

## Library Options

### `i18next` + `react-i18next` ← **Recommended**

- **Packages:** `i18next`, `react-i18next`
- **Approach:** `i18next` is the translation engine; `react-i18next` provides `useTranslation`, `Trans`, and `I18nextProvider`. JSON translation files; namespace support; typed keys via `CustomTypeOptions`.
- **Expo:** No native dependencies. Works in managed workflow with zero config beyond the `locales/` directory.
- **Strengths:** Best-in-class TypeScript integration, robust plural rules (essential for German), large community, thorough documentation.
- **Weaknesses:** Slightly more setup than lighter alternatives — requires wiring `initReactI18next` and declaring resource types.

### `expo-localization` (locale detection only, not translation)

- **Package:** `expo-localization`
- **Role in this stack:** Locale detection only — reads device locale and Android 13+ per-app locale preference. Used alongside `i18next`, not instead of it.
- **Expo:** First-party Expo package; managed workflow.
- **Note:** Provides translation helpers via `i18n-js` integration, but `i18n-js` has weaker TypeScript support and no namespace system. Recommended only as the detection layer.

### `lingui`

- **Packages:** `@lingui/react`, `@lingui/core`, `@lingui/macro`
- **Approach:** Compile-time extraction — strings are written as `<Trans>` JSX or `t` tagged templates in source; a CLI extracts them into `.po` files.
- **Expo:** Works in managed workflow but requires Babel macro configuration (`babel-plugin-macros`).
- **Strengths:** Strings live in source, so untranslated keys are impossible; ICU plural syntax.
- **Weaknesses:** The Babel macro requirement adds build complexity; `.po` file format is less familiar than JSON for contributors.

### `i18n-js` (standalone)

- **Package:** `i18n-js`
- **Approach:** Lightweight JSON-based translation with `expo-localization` for locale detection.
- **Expo:** Compatible.
- **Weaknesses:** No built-in TypeScript key inference; plural support is manual; smaller community than i18next. Not recommended for a growing translation corpus.
